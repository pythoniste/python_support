# 05 — `base64` : encoder des octets en ASCII

## 5.1 Le problème que base64 résout

Beaucoup de canaux ne savent transporter **que des caractères ASCII
imprimables** : un fichier `.env`, un en-tête HTTP, une valeur dans un
JSON, un courriel en pièce jointe. Or les vraies données — clés
binaires, images, archives — sont des **octets quelconques**, qui
contiennent souvent des valeurs hors plage imprimable (`\x00`, `\xff`…).

**base64** est un encodage qui transforme une suite d'octets
arbitraires en une chaîne **ASCII** sans guillemets, sans saut de
ligne, sans caractère exotique. À l'arrivée, on décode pour retrouver
les octets d'origine, bit pour bit.

```text
b"\x01\x02\xff\xfe"   --base64-->   "AQL//g=="
```

## 5.2 Ce que base64 N'est PAS

**base64 n'est pas un chiffrement.** N'importe qui peut décoder une
chaîne base64 sans clé, sans secret, sans rien. Le seul rôle de
base64 est de **transporter** des octets bruts à travers des canaux
qui n'aiment que l'ASCII.

```python
import base64
print(base64.b64decode("Ym9uam91cg=="))   # b'bonjour'
```

Si on veut protéger des données, il faut un **vrai** chiffrement
(`cryptography`, `pynacl`…), pas un simple encodage. Un mot de passe
encodé en base64 dans un fichier de configuration est **en clair**
pour qui sait lire la documentation.

## 5.3 Les quatre fonctions à connaître

```python
import base64

octets = b"\x01\x02\xff\xfe"

# Standard : peut contenir +, /, =
encode = base64.b64encode(octets)         # b'AQL//g=='
print(encode)
decode = base64.b64decode(encode)         # b'\x01\x02\xff\xfe'
print(decode)

# URL-safe : remplace + par -, / par _ (compatible URL/JSON)
encode_url = base64.urlsafe_b64encode(octets)   # b'AQL__g=='
decode_url = base64.urlsafe_b64decode(encode_url)
```

| Fonction                          | Caractères de sortie       | Cas d'usage          |
|-----------------------------------|----------------------------|----------------------|
| `b64encode(bytes_)`               | `A-Z a-z 0-9 + / =`        | Fichier, courriel    |
| `b64decode(s)`                    | `bytes` d'origine          | Décodage standard    |
| `urlsafe_b64encode(bytes_)`       | `A-Z a-z 0-9 - _ =`        | URL, en-tête HTTP    |
| `urlsafe_b64decode(s)`            | `bytes` d'origine          | Décodage URL-safe    |

Toutes ces fonctions travaillent en **`bytes`** : `b64encode` prend des
`bytes` en entrée et **renvoie des `bytes` ASCII**. Pour obtenir une
`str` confortable à stocker dans un JSON ou un `.env`, on décode
ensuite en ASCII :

```python
texte = base64.b64encode(octets).decode("ascii")
print(type(texte), texte)        # <class 'str'> AQL//g==
```

## 5.4 Le bourrage `=`

base64 travaille par paquets de 3 octets en entrée → 4 caractères en
sortie. Quand l'entrée n'est pas un multiple de 3, on **complète** avec
le caractère `=` :

| Octets en entrée | Sortie base64    |
|------------------|------------------|
| `b"abc"` (3)     | `"YWJj"`         |
| `b"ab"` (2)      | `"YWI="`         |
| `b"a"` (1)       | `"YQ=="`         |

Le `=` final n'apporte aucune information : c'est juste pour respecter
la taille fixe. Certaines API web suppriment ces `=` pour gagner
quelques octets ; il faut alors les remettre avant `b64decode`, ou
utiliser une variante qui s'en passe.

## 5.5 Standard ou URL-safe ?

La version standard utilise les caractères `+` et `/`. Or :

- `+` a une signification dans les URL (espace) ;
- `/` est le séparateur de chemin.

Donc dès qu'une donnée doit voyager dans une URL, un nom de fichier ou
un en-tête HTTP, on utilise `urlsafe_b64encode` / `urlsafe_b64decode`,
qui produisent les mêmes octets en sortie de décodage mais utilisent
`-` et `_` à la place.

## 5.6 Cas typiques

### Stocker une clé binaire dans un fichier `.env`

```python
import base64, secrets

cle = secrets.token_bytes(32)
texte = base64.b64encode(cle).decode("ascii")
# Ecrit dans .env :  CLE_API=tQq3...==
```

À la relecture :

```python
import os, base64
cle = base64.b64decode(os.environ["CLE_API"])
```

### Embarquer un fichier binaire dans un JSON

```python
import base64, json

with open("logo.png", "rb") as f:
    octets = f.read()

payload = {"nom": "logo.png",
           "contenu": base64.b64encode(octets).decode("ascii")}
print(json.dumps(payload))
```

À la réception, on inverse :

```python
recu = json.loads(...)
octets = base64.b64decode(recu["contenu"])
```

## À retenir

- base64 = **encodage**, jamais un chiffrement.
- Transforme des octets quelconques en chaîne ASCII transportable
  (fichier `.env`, JSON, URL, en-tête HTTP).
- `b64encode(bytes_)` renvoie des `bytes` ASCII ; on fait `.decode("ascii")`
  pour une `str`.
- `urlsafe_b64encode/decode` pour ce qui doit voyager dans une URL.
- Le `=` final est juste du bourrage ; certaines API le suppriment.
- L'opération est entièrement réversible : `b64decode(b64encode(x)) == x`.
